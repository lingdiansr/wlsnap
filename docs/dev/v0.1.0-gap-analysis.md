## 一、v0.1.0 目标回顾

**核心目标**: CLI 截图闭环 — 用户通过命令行执行截图后，可直接保存到文件、复制到剪贴板、输出到 stdout 或调用外部程序。无 GUI 编辑器，无 Pin，无长截图。

**达成条件**（来自 `03-roadmap.md`）:
- T1~T5 + T8 + T9 + T16 + T17 完成
- 手动验证: `--full` / `--screen` / `--full-all` / `--area` 均能输出正确图像
- `--clipboard` 可用，`--stdout` 输出 PNG 格式正确

## 三、关键差距详解

### 3.1 架构层面差距

#### 1. 截图后端抽象层不完整

**设计文档要求**:
```rust
#[async_trait]
pub trait CaptureBackend: Send + Sync {
    fn capabilities(&self) -> CaptureCapabilities;
    async fn capture_current_screen(&self, cursor: bool) -> Result<CapturedFrame>;
    async fn capture_all_screens(&self, cursor: bool) -> Result<CapturedFrame>;
    async fn capture_region(&self, region: LogicalRect, output: &OutputInfo, cursor: bool) -> Result<CapturedFrame>;
    async fn capture_window(&self, window: &WindowInfo, cursor: bool) -> Result<CapturedFrame>;
}
```

**当前实现**:
- 无 `CaptureBackend` trait
- 直接调用 `capture::output::capture_current_screen()` / `capture_all_screens()`
- 无 `CapturedFrame` 类型，使用 `CapturedImage { image, source_output }`
- 无窗口截图能力

**影响**: 低。v0.1.0 仅支持 wlr-screencopy，单后端直接调用更简单。抽象层在 v0.4.0（Portal 支持）时才必要。

#### 2. 状态机过度简化

**设计文档要求**:
- `Idle → SelectingRegion → Capturing → Editing → ChoosingAction → Idle`
- 支持 `Esc` 取消、状态回退

**当前实现**:
- 状态机定义为枚举，但实际仅使用 `Idle → Capturing → Idle`
- `--post edit` / `--post ask` 直接映射为 `Save`，不进入编辑器
- 无 `SelectingRegion` / `SelectingWindow` / `Editing` / `Scrolling` 实际逻辑

**影响**: 低。符合 v0.1.0 "无 GUI 编辑器" 的明确约束。

#### 3. 单实例缺失

**设计文档要求**:
- T7: Unix domain socket 绑定 + 命令转发
- 避免多个选区窗口冲突

**当前实现**:
- 完全未实现
- 每次运行都是独立进程

**影响**: 中。v0.1.0 纯 CLI 模式下冲突概率低，但 `--area` 交互选区时可能产生多个窗口。

### 3.2 功能层面差距

#### 4. `--area` 交互选区的 headless 路径

**设计文档要求**:
> "`--area` 在 v0.1.0 中通过硬编码区域参数或后续通过外部工具选区后传入，不启动 GUI 选区窗口。"

**当前实现**:
- `--area x,y,w,h` → headless 裁剪 ✅
- `--area`（无坐标）→ 启动 layer-shell / eframe 选区 ⚠️

**差距**: 设计说 v0.1.0 不启动 GUI 选区，但实际实现了。这是**正向偏差**（功能比设计更强）。

#### 5. 多屏拼接算法

**设计文档要求**:
> "计算所有 `logical_geometry` 的并集作为全局 canvas 尺寸...将各图像按 `logical_geometry.position` 拼入 canvas。"

**当前实现**:
- 已改为物理坐标紧密拼接（去掉间隙）
- 同逻辑位置的屏幕取最大宽高，不同位置紧密排列

**差距**: 实现比设计更优，但偏离了原始设计。紧密拼接是用户明确要求的功能增强。

#### 6. 配置系统完整度

**设计文档要求**:
```rust
pub struct Config {
    pub general: GeneralConfig,
    pub editor: EditorConfig,
    pub pin: PinConfig,
    pub scrolling: ScrollingConfig,
    pub shortcuts: ShortcutsConfig,
    pub advanced: AdvancedConfig,
}
```

**当前实现**:
- `general` + `advanced` 已实现 ✅
- `editor` / `pin` / `scrolling` / `shortcuts` 有定义但无实际使用 ⚠️

**影响**: 低。v0.1.0 不涉及编辑器/Pin/长截图，这些配置项为后续版本预留。

### 3.3 质量层面差距

#### 7. 测试覆盖

| 模块 | 当前测试 | 设计期望 | 差距 |
|------|---------|---------|------|
| `backend/wlr.rs` | 3 个测试（BGRA 转换、无 Wayland 错误） | 集成测试：真实捕获 | 无真实 Wayland 环境的 CI 测试 |
| `capture/output.rs` | 3 个测试（无 Wayland、物理坐标） | 多屏拼接场景测试 | 缺少多屏 mock 测试 |
| `image_engine/` | 14 个测试 | 完整覆盖 | 基本覆盖 |
| `ui/` | 0 个测试 | 交互测试 | UI 测试困难，符合预期 |
| **总计** | **147 个测试** | - | 单元测试充分，集成测试不足 |

#### 8. CLI 诊断参数

**设计文档要求**:
- `--list-outputs`: 彩色表格输出
- `--debug-protocol`: 协议探测详情

**当前实现**:
- `--list-outputs` ✅ 已实现（简单列表）
- `--debug-protocol` ✅ 已实现（打印 probe 结构体）
- 但输出格式非彩色表格，为纯文本

**影响**: 低。功能可用，美观度不足。

## 五、结论

### v0.1.0 发布 readiness: **基本就绪，需补充文档**

**核心功能**: 100% 完成。CLI 截图闭环（捕获 → 保存/剪贴板/stdout）完全可用。

**与设计的偏差**:
1. **正向偏差**: `--area` 交互选区提前实现（设计说 v0.1.0 不做）
2. **正向偏差**: 多屏拼接从逻辑坐标改为物理坐标紧密拼接（体验更好）
3. **架构简化**: 无 CaptureBackend trait（v0.4.0 多后端时才需要）

**发布前需完成**:
1. [x] 编写用户级 README（安装、基本用法、快捷键配置示例）
2. [x] 更新设计文档移除 `--exec` 和 `post_capture` 引用
3. [x] `--version` 已由 clap 自动生成

**可延后到 v0.2.0+**:
- 单实例守护（T7）
- 用户文档完整化
- AUR/PKGBUILD
- 单元测试覆盖 `ui/` 模块
